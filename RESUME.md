# 品牌矿工 Brand Miner — 简历描述 + 面试 FAQ

## 一句话描述

面向非品牌电商商家的智能选品决策系统，输入任意品类，自动输出数据驱动的选品报告。

---

## 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 后端框架 | FastAPI + uvicorn | REST API，原生 async，自动生成 API 文档 |

## 项目架构

```
Vue 3 前端（brand-miner.pages.dev）
  ├── Home.vue         — 品类输入 + 约束条件 + 历史记录

FastAPI 后端（Railway）
  ├── api/routes/        — 8 个端点（health/analysis/search/price/report）
  ├── analysis_orchestrator.py — 分析编排引擎（463 行，核心逻辑）

  ├── ai/                — LLM 客户端 + 报告生成器

```

## 项目价值

**对非品牌商家**：选品从"靠感觉跟风"变成"数据驱动决策"。入场前就知道这个品类有没有品牌壁垒、市场在涨还是跌、用户抱怨什么、定什么价有钱赚——分析一个品类从半天手动翻评论拉 Excel 压缩到 20 秒。

**对公司/团队**：选品不再拍脑袋，四维评分量化可比较、可追溯、可迭代。同一个框架对比不同品类，哪个更值得做从"我觉得"变成"数据说"。汇报时拿的是一份完整分析报告，不是一张 Excel。

**对面试**：一个涵盖全栈、异步编程、LLM 集成、实时推送、部署运维的完整闭环项目。能讲清楚"为什么这么做"的项目，比罗列十个只搭脚手架的 Demo 有用十倍。

## STAR 法则描述

**Situation（背景）**

非品牌电商商家（白牌卖家）在选品时缺乏数据支撑，传统选品依赖个人经验和主观判断，失败率高。市面上没有面向这一群体的低成本选品决策工具。

**Task（任务）**

构建一个智能选品引擎，让用户只需输入品类关键词，系统自动分析市场数据、识别细分机会、输出可执行的选品建议，降低选品决策门槛。

**Action（行动）**

- 设计四维评分模型：品牌真空度（识别垄断缺口）、自然增长信号（排除广告干扰）、用户痛点密度（从差评中提炼需求）、价格利润空间
- 实现三层数据源降级架构：阿里天池脱敏数据 → 京东轻量爬虫 → 结构化模拟数据，保证演示环境永不断线
- 基于 Server-Sent Events 实现实时分析推送，前端 ThinkingStream 组件逐条消费事件，还原 AI 逐步思考的交互体验
- 接入 DeepSeek API 流式生成决策报告，不可用时自动降级到规则模板
- asyncio.gather 实现多细分方向并行分析，全链路异步非阻塞

**Result（结果）**

- 输入任意品类（如"宠物""家居""户外"），系统自动识别 6-8 个细分方向并完成四维评分
- 完整分析流程约 15-25 秒，SSE 事件逐条推送到前端，用户可感知"系统正在思考"
- AI 报告包含 7 个结构化区块（市场概况 / 核心推荐 / 竞争机会 / 需求洞察 / 定价建议 / 风险提示 / 行动建议）
- 项目已部署到 Railway + Cloudflare Pages，公网可访问

## 面试 FAQ — 可能被追问的问题

### Q：数据是真实的吗？

系统有三层数据源。天池脱敏数据集是阿里开放的真实电商数据（约 10GB CSV），京东爬虫可以抓取实时搜索结果（30条/次，24h缓存），演示环境用的是结构化模拟数据。模拟数据遵循真实市场规律：品牌二八分布（20% 品牌占据 80% 销量）、价格正态分布、销量指数衰减，并且覆盖 10 大品类知识库，保证品类关键词的语义合理性。

### Q：评分模型怎么设计的？

四个维度加权求和，权重分别是品牌真空度 0.35、增长信号 0.30、痛点密度 0.20、价格利润 0.15，参考了选品决策中各因素的重要程度。用户输入约束条件会动态调整权重——比如选择"白牌切入"策略，品牌真空度权重从 0.35 提升到 0.40；选择"快速起量"，增长信号权重从 0.30 提升到 0.35；选择"差异化竞争"，痛点密度权重从 0.20 提升到 0.25。

### Q：SSE 和 WebSocket 为什么选 SSE？

这个场景是单向推送——服务端推数据给客户端，不需要双向通信。SSE 比 WebSocket 更轻量：原生 EventSource API 支持自动重连、HTTP 协议无需额外握手、服务端实现更简单。而且 SSE 天然适配这个场景的"只读"语义——前端只需要消费事件流，不需要向服务端发送命令。

### Q：LLM 接入有什么挑战？

主要是两个：一是流式输出要通过 SSE 透传给前端——DeepSeek 的 chat_stream 是 async generator，需要桥接到 SSE 的 asyncio.Queue，再把 token 逐条包装成 `ai_report_chunk` 事件推送到前端；二是 LLM 不稳定——API Key 未配置、429 限流、网络超时等情况必须有完整的 fallback 机制。我的处理方式是：所有异常静默捕获，永不抛异常，不可用时自动降级到结构化模板报告（基于实际评分数据填充的 7 区块报告），保证演示不中断。

### Q：如果要做成真实产品下一步怎么做？

三个方向：第一，接入真实电商数据 API——京东联盟（jd.union.open）和淘宝开放平台（taobao.tbk）都有选品相关的 API，可以直接获取实时销量和价格数据；第二，用户系统——保存历史分析记录，支持对比不同品类的分析结果，建立用户的选品决策档案；第三，评分模型验证——用 3-6 个月的真实销售数据做回测，验证权重参数是否与实际市场表现一致，迭代优化模型。

### Q：技术架构上最有挑战的部分？

最有挑战的是 SSE 流式推送的完整链路：从 AnalysisOrchestrator 的异步分析 → asyncio.Queue 事件队列 → sse-starlette 的 EventSourceResponse → 前端 EventSource API 消费。这条链路涉及 async generator、队列满丢弃策略、连接断开的资源清理、心跳保活等多个细节。特别是加入 AI 报告流式输出后，token 级别的 `ai_report_chunk` 事件需要批量推送（每 50 字一组）以避免队列溢出。

### Q：你在这个项目里学到了什么？

三个收获：第一，SSE 在实时推送场景下比 WebSocket 更适合，实现成本低且自带重连；第二，async/await 全链路异步在 Python 中的实践——从 FastAPI → SQLAlchemy 2.0 async → asyncio.gather 并行，整个链路没有同步阻塞；第三，LLM 接入必须有"不依赖 LLM 也能用"的底线——任何 AI 服务都是不可靠的，fallback 不是备选方案，是必选方案。
